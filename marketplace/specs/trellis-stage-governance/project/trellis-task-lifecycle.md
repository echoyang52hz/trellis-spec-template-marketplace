# Trellis 任务生命周期

> 定义什么时候创建任务、规划深度如何选择，以及任务启动、收尾和归档顺序。

## 任务创建边界

默认不创建 Trellis 任务的情况：

- 用户明确要求直接修改或先不走流程；
- 小范围文案、链接、错别字、格式或状态修正；
- 单条已确认规则补充，且无需过程留痕；
- 纯问答、查状态、解释流程或仅读取已有文档。

应创建 Trellis 任务的情况：

- 用户明确要求按 Trellis 流程推进；
- 工作已纳入阶段计划、里程碑或任务清单；
- 新功能、缺陷修复、重构、跨模块变更或涉及业务代码实现；
- 需要 PRD、设计说明、执行计划、验收标准、归档或 journal；
- 预计跨多个提交、多个会话或需要多人 / 多智能体协作。

## 规划产物深度

| 情况 | 规划产物 |
|------|----------|
| 单点、小范围、需求和验收明确 | `prd.md` 可以足够 |
| 阶段计划内任务、审查型任务、需要 MR 后同步阶段台账的任务 | `prd.md` + `design.md` + `implement.md` |
| 跨多个文件、多个交付物或后续任务会引用结论 | `prd.md` + `design.md` + `implement.md` |

`prd.md` 写目标、约束和验收标准；`design.md` 写边界、结构、取舍和风险；`implement.md` 写执行清单、验证命令、提交边界和回滚点。

## 启动前检查

- [ ] 已读取当前任务的 `prd.md`；
- [ ] 如存在 `design.md` 和 `implement.md`，已一并读取；
- [ ] 已读取相关 `.trellis/spec/**`；
- [ ] 任务分支、目标分支和阶段编号符合项目规则；
- [ ] 如用户明确授权偏离常规流程，已写入任务 artifact。

## 标准收尾顺序

Trellis 驱动任务推荐顺序：

```text
work commit -> finish-work -> archive commit -> journal commit -> push
-> create MR / PR -> docs progress commit -> push -> merge MR / PR
```

执行要求：

- 工作产物提交必须先于 finish-work；
- finish-work 产生的归档和 journal 应进入同一工作链路；
- 阶段任务必须等 MR / PR 编号确定后，再同步阶段执行台账；
- 不要先合并只含工作产物的 MR / PR，再用另一个变更补归档和 journal。

## Inline 模式

当智能体环境支持 inline 模式时，主会话可以直接读取规范、实现变更、运行检查和完成提交，不默认派发实现或检查子代理。

要求：

- 开始实现前读取任务 artifact 和相关 spec；
- 完成实现后运行项目要求的检查；
- 主会话对最终结果负责；
- 即使使用了额外协作者或子代理，主会话也要独立核验结果。
