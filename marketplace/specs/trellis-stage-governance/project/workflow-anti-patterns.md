# 工作流反例

> 用反例说明为什么任务、提交、归档和阶段台账的顺序不能混在一起。

## 反例：活动任务目录混入 work commit

偏差：

- work commit 包含 `.trellis/tasks/<active-task>/` 下的 PRD、task.json、design、implement 或 jsonl 文件。

风险：

- 活动任务目录是过程记录，不是工作产物；
- 归档后容易产生额外删除提交；
- 审阅者难以判断哪些文件才是本次真正产物。

正确做法：

- work commit 只提交主要产物和必要 spec；
- 活动任务目录由 finish-work 或 archive 流程处理；
- archive commit 只承担归档职责。

## 反例：先同步阶段台账，再完成 Trellis 收尾

偏差：

- work commit 后立即把阶段执行台账标记为完成；
- 此时任务尚未归档，journal 尚未记录，MR / PR 编号尚未确定。

风险：

- 阶段台账记录的是半成品状态；
- 后续审计无法把台账、归档、journal 和 MR / PR 对齐。

正确顺序：

```text
work commit -> finish-work -> archive commit -> journal commit -> push
-> create MR / PR -> docs progress commit -> push
```

## 反例：把任务书当成进度台账

偏差：

- 普通阶段子任务收尾时修改阶段任务书；
- 在任务书中写当前状态、下一步、MR / PR、提交或完成记录。

风险：

- 稳定规划契约和滚动事实混在一起；
- 阶段范围和任务拆分容易被无意改写；
- 多个入口之间出现状态漂移。

正确做法：

- 阶段任务书保存稳定规划；
- 阶段执行台账保存滚动状态；
- 修订任务书必须有明确授权，并与 docs progress commit 分开处理。

## 反例：阶段编号不一致

偏差：

- 任务标题、slug、分支名、MR / PR 标题中的阶段号或任务号不一致。

风险：

- 阶段计划、Trellis 归档、提交历史和 MR / PR 无法稳定互相检索；
- 后续协作者难以确认某个变更属于哪个阶段任务。

正确做法：

- 阶段任务统一使用 `Pxxx-Txxx` 口径；
- `T000` 只用于阶段规划；
- `T001+` 用于阶段内实现、验证和收束任务；
- 非阶段任务不强行套用阶段编号。

## 反例：把可复用规则只写在单个任务计划里

偏差：

- 某条流程警戒只写在当前任务的 `implement.md`，没有沉淀到 `.trellis/spec/**`。

风险：

- 当前任务归档后，新任务默认读不到这条规则；
- 上下文压缩、换平台或换协作者后容易复发同类问题。

正确做法：

- 任务级计划写本任务的具体执行步骤；
- 已多次验证的通用规则应沉淀到项目级 spec；
- 新任务开始前通过开发前规范读取这些规则。
