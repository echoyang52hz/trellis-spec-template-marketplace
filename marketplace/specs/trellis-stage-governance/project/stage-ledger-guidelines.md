# 阶段执行台账规范

> 阶段执行台账是阶段当前状态入口和过程事实记录，不是阶段规划契约。

## 文档职责

阶段执行台账用于记录滚动事实：

- 当前状态；
- 下一步任务；
- 启动下一任务前必须读取的材料；
- 已完成任务记录；
- MR / PR、分支、主要 work commit 和 Trellis 归档 PRD；
- 阶段关键问题处理记录；
- docs progress commit 维护清单。

执行台账不得改写阶段范围、任务拆分或验收标准。规划变更应回到阶段任务书修订流程。

## 完成记录规则

阶段子任务完成后，执行台账应记录：

- 任务编号和任务名；
- Trellis 归档 PRD 路径或链接；
- MR / PR 编号或链接；
- 工作分支；
- 主要 work commit；
- 关键验证结论；
- 如处理了阶段核心问题，记录处理证据。

“主要提交”只记录 work commit，不记录 archive commit、journal commit 或 docs progress commit。

## 同步时机

阶段执行台账应在 MR / PR 编号确定后同步。

推荐顺序：

```text
work commit -> finish-work -> archive commit -> journal commit -> push
-> create MR / PR -> docs progress commit -> push
```

不要在 MR / PR 编号尚未确定时，把任务标为最终完成或写入“待创建 MR”这类半成品状态。

## 当前状态与下一步

台账中的当前状态应帮助协作者快速判断：

- 阶段现在处于哪个任务之后；
- 下一步应启动哪个任务；
- 启动前应读取哪些任务书章节、spec 或历史任务；
- 哪些问题已经关闭，哪些仍需后续处理。

外围 README、路线图或架构文档可以链接执行台账，但不应复制具体下一任务、MR / PR、分支或完成台账，以免多处状态漂移。

## 推荐骨架

```markdown
# 阶段<编号>执行台账

## 文档职责

## 维护规则

## 当前状态

## 下一步任务

## 当前任务入口

## 完成台账

## 关键问题处理记录

## Docs Progress 检查清单
```
