# 阶段任务书规范

> 阶段任务书是阶段规划契约，不是滚动进度台账。

## 文档职责

阶段任务书用于记录一个阶段的稳定规划：

- 阶段目标和为什么现在要做；
- 阶段范围和不包含范围；
- 问题分级或处置规则；
- 关键能力、交付维度或验收维度；
- 任务清单、任务拆分和依赖关系；
- 每个任务的定位、主要产出和验收依据；
- 信息来源和后续阶段候选输入。

任务书不记录当前状态、下一步任务、MR / PR 编号、分支、提交、归档 PRD 或 journal。

## T000 规则

每个阶段用于规划、任务拆分和生成阶段文档的启动任务，建议编号为 `T000`。

`T000` 负责：

- 收集上一阶段或启动输入；
- 明确阶段目标、范围和不包含范围；
- 拆分 `T001+` 任务；
- 创建或更新阶段任务书；
- 创建阶段执行台账初始版本；
- 明确后续任务启动时应读取的材料。

`T000` 不占用后续实现、验收或收束任务编号。

## T001+ 规则

`T001+` 用于阶段内可独立验收的任务，例如实现、修复、验证、文档沉淀或收束。

每个任务应能回答：

- 任务目标是什么；
- 任务范围和非范围是什么；
- 依赖哪些前置任务或输入；
- 主要产物是什么；
- 如何验收；
- 完成后是否需要同步阶段执行台账。

## 任务书修订

任务书在 `T000` 完成后默认视为稳定规划契约。普通阶段子任务收尾时不应顺手修改任务书。

如确需修订任务书，应满足：

- 用户或团队明确授权；
- 修订原因清楚；
- 与执行台账进度同步分开处理；
- 使用独立 commit 或独立 MR / PR，避免混入普通任务收尾。

## 推荐骨架

```markdown
# 阶段<编号>任务书

## 文档职责

## 阶段目标

## 范围

## 不包含范围

## 问题分级或处置规则

## 关键交付维度

## 任务清单总览

## 执行顺序与依赖关系

## 各任务详细说明

## 阶段完成后的候选输入

## 信息来源
```
