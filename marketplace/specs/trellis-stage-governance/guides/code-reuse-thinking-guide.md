# 代码复用思考指南

> 新增实现前，先确认项目里是否已经存在相同或相近模式。

## 为什么要先想复用

重复实现会带来长期不一致：

- 一个地方修了 bug，另一个地方没有同步；
- 两套相似逻辑随着时间漂移；
- 后续协作者难以判断哪个实现才是标准；
- 测试和文档需要重复维护。

## 写新逻辑前

先搜索：

```bash
rg -n "关键字或函数名"
```

然后问自己：

| 问题 | 如果答案是“是” |
|------|----------------|
| 是否已有相似函数、工具或组件？ | 优先复用或扩展 |
| 是否已有相似目录结构或命名模式？ | 跟随现有模式 |
| 是否正在复制一段逻辑？ | 停下来考虑抽取共享实现 |
| 是否准备新增常量或配置？ | 先找是否已有单一来源 |
| 是否会有多个调用方使用？ | 放在合适的共享位置 |

## 什么时候抽象

适合抽象：

- 相同逻辑出现三次或更多；
- 逻辑复杂，未来容易出现 bug；
- 多个模块或多人会复用；
- 已经出现过因为重复实现导致的不一致。

不适合抽象：

- 只用一次；
- 是非常直白的一行逻辑；
- 抽象后的理解成本高于重复本身；
- 需求仍在快速变化，边界还不稳定。

## 批量修改后

- [ ] 回头检查是否漏了同类位置；
- [ ] 再搜索一次旧值、旧字段或旧路径；
- [ ] 确认测试、文档和配置是否需要同步；
- [ ] 判断是否应该沉淀为共享工具、规范或模板。
