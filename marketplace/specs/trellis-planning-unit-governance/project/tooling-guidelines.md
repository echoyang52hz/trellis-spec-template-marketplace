# 工具链规范

> 仓库级工具运行范围、环境隔离、官方 CLI 和质量门禁约定。

---

## 适用范围

当你在项目中运行测试、格式化、类型检查、构建、迁移、一次性诊断脚本、代码托管平台 CLI 或本地服务工具时，应先确认本文件中的边界。

---

## 环境隔离

通用原则：

- 优先使用项目声明的包管理器、虚拟环境、容器或工具链包装命令；
- 不要为了让本地命令通过而修改系统级运行环境；
- 不要把依赖安装到系统 Python、全局 Node.js、全局 Ruby 或其他共享运行环境，除非项目明确要求；
- 发现依赖缺失时，应优先修复项目依赖元数据、锁文件、容器镜像或本地开发环境初始化脚本；
- 一次性诊断命令也应遵守项目环境隔离，不把“临时修复”留在全局环境里。

示例：

```bash
# 推荐：使用项目托管环境
<package-manager> test
<package-manager> lint
<package-manager> typecheck

# 不推荐：为了当前任务直接污染系统环境
sudo <package-manager> install <dependency>
```

---

## 官方 CLI 优先

涉及代码托管平台、云平台、制品库或部署平台时：

- 优先使用平台官方 CLI；
- 使用前确认 CLI 已安装、已认证且作用域正确；
- 不使用非官方 CLI 处理敏感仓库、MR / PR、token、部署或权限操作，除非用户或团队明确允许；
- 认证失败时，优先改用官方 Web 界面或让用户完成认证，不把 token 写入仓库文件、任务记录或命令历史。

---

## 质量命令边界

修改代码或规范后，应按影响范围选择质量命令：

| 变更类型 | 建议检查 |
|----------|----------|
| 单纯文档或 spec | Markdown / 链接 / JSON 校验，必要时检查目录索引 |
| 单模块代码 | 该模块测试、格式化、类型检查或 lint |
| 跨模块代码 | 相关模块测试 + 项目级质量门禁 |
| 构建或部署配置 | 配置渲染、dry-run、构建上下文检查或平台官方校验 |
| 依赖或工具链 | 锁文件一致性、安装模拟、CI 等价命令 |

命令失败时，应区分：

- 本次改动引入的新失败；
- 既有环境缺失或历史失败；
- 需要用户权限、外部服务或凭据的阻塞。

不要把“无法运行”写成“已通过”。如果检查受阻，应记录阻塞原因、已执行命令和剩余风险。

---

## 构建上下文与敏感信息

修改构建、部署、打包或示例配置时，应检查：

- 构建上下文不包含任务过程目录、workspace journal、缓存、依赖目录、构建产物或本地 `.env`；
- 示例配置只包含占位值，不包含真实 token、密码、私有地址或用户个人路径；
- 复杂类型环境变量应使用项目约定的可解析格式，不创造临时分隔符协议；
- 文档、示例和实际配置的变量名、默认值、类型说明保持一致。

提交前可以使用搜索命令扫描敏感信息和旧值：

```bash
rg -n "token|password|secret|private|localhost|127\\.0\\.0\\.1|TODO" .
```

目标项目可以按自身情况扩展更精确的敏感信息扫描规则。

---

## 修改工具规则时

- 工具规范应写成触发场景、命令形式、约束规则和必要检查，而不是只写经验口号；
- 如果某条规则来自一次具体事故，应抽象成通用反例，不保留业务服务名、个人路径或内部地址；
- package / layer 级工具规则可以补充具体命令，但不应覆盖项目级环境隔离和敏感信息原则；
- 新增工具时，应同步 README、开发手册、CI 配置和 `.trellis/spec/**` 中相关入口。
