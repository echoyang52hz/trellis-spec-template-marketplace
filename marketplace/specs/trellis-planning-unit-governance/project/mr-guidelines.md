# MR / PR 规范

> 分支命名、MR / PR 标题与描述、推送和合并后清理规范。

---

## 分支规则

### 分支创建规则

默认基于主开发分支创建独立工作分支。目标项目初始化模板后，应把“主开发分支”替换为真实分支名。

| 场景 | 操作要求 |
|------|----------|
| **新功能开发** | 基于主开发分支创建 `feat/功能名` |
| **缺陷修复** | 基于主开发分支创建 `fix/问题简述` |
| **文档更新** | 基于主开发分支创建 `docs/文档范围` |
| **工具 / 配置调整** | 基于主开发分支创建 `chore/调整范围` |
| **功能组开发** | 建立主功能分支，如 `feat/payment-system`；子功能分支基于主功能分支创建并 MR / PR 回主功能分支 |

### 分支命名规范

- 分支分类必须使用 `/` 连接，例如 `docs/project-git-workflow-spec`。
- 禁止使用 `docs-project-git-workflow-spec` 这类用 `-` 代替分类分隔的命名。
- 分支名不写与项目产物无关的工具生成说明；Trellis 框架相关内容可以按事实描述。

## 计划单元编号类型

计划单元是项目中用于组织一组可追踪 Trellis 任务的流程单元。常见计划单元类型包括：

| 类型 | 前缀 | 定位 | 示例 |
|------|------|------|------|
| 阶段 | `P<三位序号>` | 项目主线演进周期 | `P001` |
| 专项 | `S<三位序号>` | 围绕单一能力域或问题域的小型阶段化推进 | `S001` |

`计划单元` 只作为规则抽象名，用于描述阶段和专项共享的流程。实际命名、标题、目录、任务书、执行台账和对外协作文档中，应使用目标项目定义的具体分类名称，不要把 `计划单元` 当作实际类型名或文档名。

计划单元内任务编号使用 `<计划单元编号>-<类型><三位序号>`。主线任务继续使用 `T`，计划外但需要 Trellis / MR / PR / 台账追踪的任务使用少量独立类型，避免占用主线任务序号。

| 类型 | 含义 | 适用场景 | 示例 |
|------|------|----------|------|
| `T` | Task 主线任务 | 计划单元任务书中已经规划并排序的任务 | `P001-T008：能力基线建设` / `S001-T001：模板 schema 设计` |
| `R` | Revision 规划修订 | 修订任务书、任务拆分、验收标准、计划路线或当前任务入口 | `P001-R001：任务拆分修订` |
| `I` | Issue 问题调查 | 已发现现象、风险或缺口，但尚未确认是否为缺陷，需要调查并给出结论 | `P001-I001：调查状态缺失问题` |
| `F` | Fix 缺陷修复 | 已确认的 bug、回归或行为错误，需要修复并验证 | `P001-F001：修复接口异常` |

执行要求：

- `T` 序号只用于计划单元任务书中明确排序的主线任务，计划外任务不得插入或占用 `T001+`；
- `R` 用于少数规划契约修订，必须有用户或团队明确授权，不得混入普通 docs progress commit；
- `I` 可以结论为不是问题、转为 `F`、转为后续主线任务，或记录为已知限制；
- `F` 应引用触发它的 `T` / `R` / `I` 或外部问题来源，并包含修复验证；
- 同一计划单元内每种类型各自独立递增，例如 `P001-T008`、`P001-R002`、`P001-I003`、`P001-F001`；
- 完成台账应按实际推进顺序或逻辑依赖顺序排列，让读者能看出“先修订 / 调查 / 修复，再继续主线任务”的关系。

## 计划单元任务命名规范

已经纳入计划单元且有明确序号的 Trellis 任务，或已经按上节分配了 `T/R/I/F` 编号的计划单元内计划外任务，应在任务标题、slug、分支和 MR / PR 标题中保留计划单元与任务编号，便于与计划单元文档、Trellis 归档和 journal 互相映射。

| 对象 | 格式 | 示例 |
|------|------|------|
| Trellis 任务标题 | `<计划单元编号>-<类型><三位序号>：<任务名>` | `P001-R001：任务拆分修订` / `S001-T000：模板化专项规划` |
| Trellis slug | `<小写计划单元编号>-<小写类型><三位序号>-<english-task-slug>` | `p001-r001-task-split-revision` / `s001-t000-template-plan` |
| 工作分支 | `<type>/<小写计划单元编号>-<小写类型><三位序号>-<english-task-slug>` | `docs/p001-r001-task-split-revision` / `docs/s001-t000-template-plan` |
| MR / PR 标题 | `[<计划单元编号>-<类型><三位序号>] <任务名>` | `[P001-R001] 任务拆分修订` / `[S001-T000] 模板化专项规划` |

执行要求：

- 编号只用于已经在计划单元中明确排序的任务；普通临时任务、探索任务和未排序维护任务不强行编号；
- 每个计划单元用于拆分任务、生成任务书和执行台账的启动任务，也属于该计划单元任务，建议编号为 `T000`；
- `T000` 只用于计划单元规划 / 任务拆分 / 任务书和执行台账生成，不占用后续实现、验收或收束任务的 `T001+` 编号；
- 已经完成并长期被引用的历史计划单元规划任务不追溯改名或改归档路径，避免破坏既有 MR / PR、commit、文档和 Trellis 归档链接；
- 命名时必须把 `计划单元` 替换为目标项目定义的具体分类名称；
- 计划单元序号和任务序号统一使用三位数字，避免数量增长后排序和搜索结果混乱；
- Trellis slug 和工作分支使用小写的 `pxxx-txxx` / `sxxx-txxx` 等形式；MR / PR 标题使用带中括号的大写展示形式 `[Pxxx-Txxx]` / `[Sxxx-Txxx]`；
- 非计划单元任务不强行套用计划单元编号。

## 禁止行为

- ❌ 直接向生产主分支提交代码；
- ❌ 直接向生产主分支推送远程；
- ❌ 在项目原有分支上直接提交新任务变更，除非用户或团队规则明确允许；
- ❌ 在分支名、commit 信息、MR / PR 标题或描述中出现与项目产物无关的工具生成说明；
- ❌ 未经用户或团队明确允许，使用非官方代码托管平台 CLI 创建或管理 MR / PR。

## MR / PR 创建

### GitLab

仓库远端托管在 GitLab 时，优先使用 **Merge Request (MR)**。如果官方 `glab` CLI 已安装并认证，可以使用：

```bash
glab mr create --target-branch <main-development-branch> --source-branch <work-branch>
```

如 `glab` 不可用或未认证，使用 GitLab Web 界面创建 MR。

### GitHub

仓库远端托管在 GitHub 时，使用 **Pull Request (PR)**。如果官方 `gh` CLI 已安装并认证，可以使用：

```bash
gh pr create --base <main-development-branch> --head <work-branch>
```

如 `gh` 不可用或未认证，使用 GitHub Web 界面创建 PR。除非用户或团队明确允许，禁止使用非官方代码托管平台 CLI。

## MR / PR 标题

MR / PR 标题是审阅入口，不等同于 commit 标题。Commit 标题可以按提交规范使用 Gitmoji 开头；MR / PR 标题通常不使用 Gitmoji。

计划单元内任务的 MR / PR 标题使用固定前缀：

```text
[<计划单元编号>-<类型><三位序号>] <任务名>
```

示例：

```text
[P001-T011] 任务执行链路闭环
[P001-R001] 任务拆分修订
[P001-I001] 调查状态缺失问题
[P001-F001] 修复接口异常
[S001-T000] 模板化专项规划
```

非计划单元任务不强行使用计划单元前缀，使用简洁标题即可。

## MR / PR 描述

MR / PR 描述应作为审阅者理解本次变更的入口。描述重点应放在本次任务的业务、代码、文档或规范产出上，而不是机械罗列 Trellis 归档、journal 等流程性提交。

推荐模板：

```markdown
## 概述

- 本次变更解决什么问题；
- 本次变更的主要产出是什么；

## 主要变更

- 变更点 1；
- 变更点 2；

## 验证

- [ ] 已运行 <命令或检查>；
- [ ] 已人工确认 <关键行为>；

## 影响范围

- 影响哪些模块、文档、部署或使用方；

## 关联

- Trellis 归档 PRD：<链接>；
- 计划单元台账或计划文档：<链接>；
- 主要提交：<链接>；
```

计划单元内任务的描述要求：

- “概述”和“主要变更”聚焦具体计划单元任务本身；
- `chore(task): archive ...`、`chore: record journal` 等提交通常不作为概览重点展开；
- 计划单元台账中的“主要提交”只记录主要 work commit，不记录 archive、journal 或 docs progress commit。

## 合并后的分支清理

MR / PR 合并后应清理已完成的工作分支。

### 清理本地分支

```bash
git checkout <main-development-branch>
git pull --ff-only
git branch --delete <work-branch>
```

### 清理远端分支

优先在 MR / PR 合并时勾选删除源分支。若需要手动清理：

```bash
git push origin --delete <work-branch>
```
