# Git 工作流总入口

> 项目级 Git 工作流总入口。先读这里，再按场景进入专题文件。

---

## 概览

项目应通过工作分支和代码托管平台的 MR / PR 合并变更。目标项目初始化本模板后，应把主开发分支名称、代码托管平台、官方 CLI 和默认目标分支改成真实情况。

如果你正在做以下事情，除了本文件，还应继续读取：

| 场景 | 继续读取 |
|------|----------|
| 准备提交代码，判断哪些文件该进 work commit | [提交边界](./commit-boundaries.md) |
| 准备推分支、写 MR / PR 标题和描述、合并后清理分支 | [MR / PR 规范](./mr-guidelines.md) |
| 准备创建 / 结束 Trellis 任务，或处理计划单元同步 | [Trellis 任务生命周期](./trellis-task-lifecycle.md) |
| 想理解为什么某些顺序不对、某些提交会多出噪音 | [工作流反例](./workflow-anti-patterns.md) |

---

## 核心原则

1. **禁止直接向生产主分支提交或推送。**
2. **默认基于主开发分支创建独立工作分支；只有用户或团队规则明确允许时，才直接在现有分支上修改。**
3. **所有合并优先通过代码托管平台的 MR / PR 完成。**
4. **提交、分支、MR / PR 标题和描述中禁止出现与项目产物无关的工具生成说明；Trellis 框架相关内容可以按事实描述。**
5. **计划单元内任务必须把 work commit、archive commit、journal commit 和 docs progress commit 分开处理；计划单元进度默认同步到执行台账。**
6. **`.trellis/spec/**` 属于 code-spec，可随相关工作提交；`.trellis/tasks/<active-task>/` 属于活动任务过程文件，不得混入 work commit。**

## 快速检查

### 提交前

- [ ] 提交信息符合项目约定。
- [ ] 当前分支符合项目工作流，或已有明确授权。
- [ ] 已确认 work commit 不包含 `.trellis/tasks/<active-task>/`。
- [ ] 如有相关 `.trellis/spec/**` 沉淀，已随 work commit 一起提交。

### 创建 MR / PR 前

- [ ] 已推送工作分支。
- [ ] 已确认 MR / PR 标题是否需要计划单元编号。
- [ ] 已按需要补齐 MR / PR 描述中的概述、主要变更、验证、影响范围和关联。

### 合并后

- [ ] 更新本地主开发分支。
- [ ] 删除本地已合并分支。
- [ ] 确认远端源分支已删除，或按平台规则手动删除。

## 拿不准时

如果你发现自己正在思考下面这些问题，说明不该只停留在本文件：

- “这个文件该不该进 work commit？”
- “现在该先 finish-work 还是先开 MR / PR？”
- “计划单元任务书或执行台账现在能不能改？”
- “为什么这次会多一个清理活动目录提交？”

这时直接跳到对应专题文件，不要只靠记忆执行。
