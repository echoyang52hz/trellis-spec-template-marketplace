# 跨层思考指南

> 当一个变化跨多个层级、模块、角色或文档时，先画清楚数据和责任如何流动。

## 常见问题

跨层问题通常不是某一层写错，而是边界没有说清楚：

- 上游输出格式和下游预期不一致；
- 某个字段在中间层被丢失或误改；
- 多层重复校验同一规则，但细节不同；
- 文档、接口、实现和测试各说各话。

## 实现前先画流转

用简单链路描述数据或状态：

```text
输入 -> 转换 -> 存储或传递 -> 读取 -> 展示或消费
```

每个箭头都问：

- 当前格式是什么；
- 谁负责校验；
- 失败时如何表达；
- 下游是否依赖这个细节；
- 有没有多个消费者。

## 边界检查

| 边界 | 常见风险 |
|------|----------|
| 接口与服务 | 字段缺失、类型不一致、错误码不一致 |
| 服务与存储 | 空值、默认值、格式转换、迁移兼容 |
| 后端与前端 | 序列化格式、时间格式、枚举值、分页结构 |
| 组件与组件 | 属性形状、事件语义、加载和错误状态 |
| 文档与实现 | 文档承诺和真实行为不一致 |

## 契约要写清

跨层变化至少应明确：

- 输入格式；
- 输出格式；
- 错误或异常形态；
- 默认值和空值处理；
- 兼容旧数据或旧调用方的策略；
- 哪一层负责校验和转换。

## 提交前检查

- [ ] 已梳理完整流转路径；
- [ ] 已识别所有边界；
- [ ] 已确认每个边界的输入输出格式；
- [ ] 已同步测试、文档或 spec；
- [ ] 已验证空值、缺失、非法输入或旧数据场景。
